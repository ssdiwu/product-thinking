# codex-assisted 模式详细指南

> 基于 ai-team 的 codex-assisted 模式改造

## 核心理念

**分离规划与执行**: product-thinking 负责规划分析任务,Codex 负责执行分析,product-thinking 负责评审质量。

```
product-thinking plan → Codex exec → product-thinking review
     (规划)              (执行)           (评审)
```

## 工作流详解

### 阶段 1: Plan (规划)

**触发**: 用户启动项目时指定 `--mode codex`

**执行步骤**:
1. 完成 Kickoff 阶段(灵魂三问 + 工具选择)
2. 为每个选定的分析工具生成任务包
3. 生成 `.product-thinking/codex-tasks.md`

**任务包结构**:

```markdown
# 产品分析任务清单

## 项目信息

- **项目名称**: XX 产品
- **目标用户**: XX
- **核心痛点**: XX
- **独特价值**: XX

## 任务 1: JTBD 分析

**目的**: 理解用户真正想完成的任务

**依赖**: 无

**输入**:
- 灵魂三问的回答
- 用户画像

**输出**:
- 文档: `phases/02-jtbd.md`
- 必须包含:
  - 三层需求分析(功能层/情感层/社会层)
  - 用户旅程图(Mermaid)
  - 核心发现总结

**分析框架**:
1. 用户"雇佣"产品来完成什么工作?
2. 功能层需求: 用户想完成什么具体任务?
3. 情感层需求: 用户希望获得什么感受?
4. 社会层需求: 用户希望在他人眼中是什么形象?
5. 绘制用户旅程图: 从触发 → 行动 → 结果

**验收标准**:
- [ ] 三层需求都已分析
- [ ] 用户旅程图已生成
- [ ] 核心发现已总结(3-5 个)
- [ ] 文档格式符合模板

**参考文档**: `references/framework/jtbd.md`

---

## 任务 2: MVP 功能审视

**目的**: 定义最小可行产品,避免过度设计

**依赖**: 任务 1 (JTBD 分析)

**输入**:
- JTBD 分析结果
- 用户的核心任务

**输出**:
- 文档: `phases/03-mvp.md`
- 必须包含:
  - P0/P1/P2 功能分层
  - MVP 验证指标
  - 功能分层图(Mermaid)

**分析框架**:
1. 核心假设是什么?(需要验证的最关键假设)
2. P0 功能: 验证核心假设的最小功能集
3. P1 功能: 增强体验但非必需
4. P2 功能: 锦上添花
5. 验证周期: 多久可以验证核心假设?

**验收标准**:
- [ ] 核心假设已明确
- [ ] P0/P1/P2 功能已分层
- [ ] MVP 验证指标已定义
- [ ] 功能分层图已生成
- [ ] 文档格式符合模板

**参考文档**: `references/framework/mvp.md`

---

## 任务 3: 场景应用分析

**目的**: 找到产品的最佳切入场景

**依赖**: 任务 1 (JTBD 分析), 任务 2 (MVP 功能审视)

**输入**:
- JTBD 分析结果
- MVP 功能列表

**输出**:
- 文档: `phases/04-scenarios.md`
- 必须包含:
  - 场景匹配矩阵
  - 场景优先级排序
  - 场景分析图(Mermaid)

**分析框架**:
1. 列举所有可能的使用场景
2. 评估每个场景的匹配度(用户需求强度 × 产品能力)
3. 评估每个场景的竞争强度
4. 选择最佳切入场景(高匹配度 + 低竞争强度)
5. 制定场景切入策略

**验收标准**:
- [ ] 场景匹配矩阵已生成
- [ ] 场景优先级已排序
- [ ] 最佳切入场景已选定
- [ ] 场景分析图已生成
- [ ] 文档格式符合模板

**参考文档**: `references/framework/scenarios.md`

---

## 任务 4: 问题发现诊断

**目的**: 验证问题的真实性和严重性

**依赖**: 任务 1 (JTBD 分析)

**输入**:
- 用户的核心痛点
- JTBD 分析结果

**输出**:
- 文档: `phases/05-diagnosis.md`
- 必须包含:
  - 问题真实性验证
  - 问题根因分析(5个为什么)
  - 问题分析树(Mermaid)

**分析框架**:
1. 问题真实性验证:
   - 这是一个真实存在的问题吗?
   - 有多少人遇到这个问题?
   - 问题的严重程度如何?
2. 问题根因分析(5个为什么):
   - 为什么会有这个问题?
   - 为什么会这样?
   - 为什么会这样?
   - 为什么会这样?
   - 为什么会这样?
3. 绘制问题分析树

**验收标准**:
- [ ] 问题真实性已验证
- [ ] 5个为什么已完成
- [ ] 问题根因已明确
- [ ] 问题分析树已生成
- [ ] 文档格式符合模板

**参考文档**: `references/framework/diagnosis.md`

## 执行说明

### 推荐命令

```bash
# 自动化执行(推荐)
codex exec "严格按照任务清单执行" -i .product-thinking/codex-tasks.md --full-auto

# 或使用管道
cat .product-thinking/codex-tasks.md | codex exec "严格按照任务清单执行"
```

### 执行约束

1. **严格按顺序执行**: 任务有依赖关系,必须按顺序执行
2. **遵循分析框架**: 每个任务都有明确的分析框架,必须遵循
3. **满足验收标准**: 每个任务都有验收标准,必须全部满足
4. **保持文档一致性**: 所有文档使用统一的格式和模板

### 文档模板

所有文档使用以下 YAML frontmatter:

```yaml
---
phase: "analysis"
tool: "jtbd"  # 或 mvp, scenarios, diagnosis
created_at: "2026-02-12T00:00:00Z"
updated_at: "2026-02-12T00:00:00Z"
status: "completed"
---
```

### 常见问题

**Q: 如果某个任务的依赖未完成怎么办?**

A: 等待依赖任务完成后再执行。Codex 会自动检测依赖关系。

**Q: 如果某个任务的验收标准未满足怎么办?**

A: 继续完善直到满足所有验收标准。product-thinking review 阶段会检查。

**Q: 如果某个任务的参考文档不存在怎么办?**

A: 使用 SKILL.md 中的分析工具说明作为参考。
```

### 阶段 2: Exec (执行)

**触发**: 用户手动执行 Codex

**用户操作**:
```bash
codex exec "严格按照任务清单执行" -i .product-thinking/codex-tasks.md --full-auto
```

**Codex 行为**:
1. 读取任务清单
2. 按顺序执行每个任务
3. 生成对应的文档
4. 更新 `project.yaml` 状态

**关键约束**:
- Codex 必须严格遵循任务清单中的分析框架
- Codex 必须满足所有验收标准
- Codex 必须保持文档格式一致性

---

### 阶段 3: Review (评审)

**触发**: 用户运行 `/product-thinking review`

**执行步骤**:
1. 读取 `.product-thinking/codex-tasks.md`(原始任务)
2. 读取所有生成的文档(实际产出)
3. 对比任务要求 vs 实际产出
4. 多维度评分:
   - 文档完整性(30%): 所有必需的文档都已生成
   - 内容质量(30%): 分析深度、逻辑一致性、数据支撑
   - 可视化图表(20%): 图表完整性、语法正确性
   - 验收标准(20%): 所有验收标准都已满足
5. 生成评审报告: `reviews/completeness-check.md`

**评审报告结构**:

```markdown

## 最佳实践

### 1. 任务清单要详细

每个任务必须包含:
- 目的(为什么做这个任务)
- 依赖(需要哪些前置任务)
- 输入(需要哪些信息)
- 输出(生成什么文档)
- 分析框架(如何分析)
- 验收标准(如何验证)

### 2. 验收标准要具体

❌ 低质量验收标准:
```
- [ ] 分析已完成
- [ ] 文档已生成
```

✅ 高质量验收标准:
```
- [ ] 三层需求都已分析(功能层/情感层/社会层)
- [ ] 用户旅程图已生成(包含触发→行动→结果)
- [ ] 核心发现已总结(3-5 个)
- [ ] 文档格式符合模板(YAML frontmatter + Markdown)
```

### 3. 评审要基于事实

评审时:
- ✅ 对比任务要求 vs 实际产出
- ✅ 检查验收标准是否满足
- ✅ 检查文档格式是否一致
- ❌ 不要猜测或假设

### 4. 改进建议要可执行

每个改进建议必须包含:
- 问题是什么
- 为什么是问题
- 如何改进
- 改进后如何验证

## 常见问题

### Q: codex-assisted 模式和 multi-agent 模式有什么区别?

**A**:
- **codex-assisted**: AI 生成任务清单,用户用 Codex 执行,AI 评审
- **multi-agent**: AI 自动派发专家 Agent 并行执行

**选择建议**:
- 需要精确控制 → codex-assisted
- 需要快速迭代 → multi-agent

### Q: 如果 Codex 执行失败怎么办?

**A**: 检查以下几点:
1. 任务清单是否完整(所有必需字段都已填充)
2. 依赖关系是否正确(前置任务是否已完成)
3. 参考文档是否存在(Codex 需要参考文档)

### Q: 如果评审不通过怎么办?

**A**: 根据评审报告的改进建议:
1. 补充缺失的内容
2. 修复错误的图表
3. 完善不足的分析
4. 重新运行 `/product-thinking review`

### Q: 可以跳过某些任务吗?

**A**: 可以,但需要:
1. 在任务清单中删除该任务
2. 更新依赖关系(如果其他任务依赖该任务)
3. 更新 `project.yaml` 中的 `selected_tools`
