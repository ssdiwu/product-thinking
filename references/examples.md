# 短例子

这份文档不是教程，而是一组短例子。

它的作用只有两个：

1. 帮新用户快速理解这个技能是怎么思考的
2. 帮模型在输出时更稳定地落到“现象 -> 判断 -> 动作 -> 验证”

不要把这些例子当模板硬套。真正重要的是其中的判别方式。

## 诊断模式

### 例子 1：好的输入长什么样

```text
我们的 SaaS 产品注册转化还行，但第二周留存很差。目标用户是 20-50 人的小团队负责人。很多人注册后第一周会导入数据，第二周就不回来了。先别给方案，先帮我判断问题更可能出在价值兑现、使用门槛，还是用户选错了。
```

为什么这是好输入：

- 现象具体，不是笼统地说“增长不好”
- 用户范围基本明确
- 已经有一个可观察的流失节点
- 明确要求先判断，不是先给方案

### 例子 2：不够好的输入如何被收束

用户原话：

```text
我们产品不太行，帮我看看怎么优化。
```

更好的收束方式：

```text
先别直接给优化建议。先帮我判断：到底是哪个环节出了问题？我目前看到的现象是，用户会注册，但很少持续回来用。
```

### 例子 3：从现象走到动作的短输出

```markdown
### 现象
- 注册不差，但第二周留存明显下滑
- 用户第一周会导入数据，之后不再回来

### 判断
- 当前更像“首次使用后没有持续价值兑现”，而不只是获客问题
- 也不排除使用门槛偏高，但优先级低于价值兑现

### 红旗
- 目前只有行为现象，没有用户原话

### 动作
- 访谈 5 个流失用户，追问他们第一次使用后为什么没再回来

### 验证
- 如果多数用户说“看完一次就够了”，说明问题更偏价值兑现
- 如果多数用户说“第二次不知道怎么继续”，说明问题更偏使用门槛
```

## 创始人模式

### 例子 1：好的输入长什么样

```text
我想做一个给独立开发者用的用户反馈工具。现在很多人用飞书、Notion 和表格凑合，我想先判断这个方向值不值得做。我的直觉是他们的痛点不是“收不到反馈”，而是“反馈散在各处，后面根本回看不了”。
```

为什么这是好输入：

- 用户不是“所有人”，而是独立开发者
- 当前替代方案被说出来了
- 痛点不是抽象口号，而是一个具体工作流问题
- 方向判断聚焦在值不值得做，不是先铺功能

### 例子 2：不够好的输入如何被收束

用户原话：

```text
我想做一个 AI 产品，帮我想想有什么机会。
```

更好的收束方式：

```text
请先别发散机会，先帮我收窄：我最可能服务的是谁，他们现在在哪个具体场景下最痛，我凭什么比现有方案更容易做对。
```

### 例子 3：从现象走到动作的短输出

```markdown
### 现象
- 目标用户暂定为独立开发者
- 他们目前用表格、文档、聊天记录分散管理反馈

### 判断
- 这个方向有可能成立，但当前真正的问题还不是“缺一个反馈入口”，而是“反馈无法被持续回看和整理”
- 现在最大的缺口是：还没有确认这个痛点是否足够强，足以让用户切换做法

### 红旗
- 用户范围仍偏大
- 还没有真实用户原话

### 动作
- 找 3 到 5 个独立开发者，复盘他们最近一次整理用户反馈的完整过程

### 验证
- 如果多个用户明确抱怨“反馈后面找不到、用不上”，方向继续成立
- 如果他们觉得现在凑合够用，方向就要收缩或重想
```

## 构建者模式

### 例子 1：好的输入长什么样

```text
我想做一个个人副项目，帮跨境卖家更快找到潜在选品机会。先别帮我做大规划，只帮我收敛第一版：一周内能做出来、能拿给 3 个卖家看的最小版本应该是什么？
```

为什么这是好输入：

- 明确是个人副项目，不是假装成熟产品
- 明确要收敛第一版，而不是泛规划
- 明确了时间边界和展示对象

### 例子 2：不够好的输入如何被收束

用户原话：

```text
我想做个很酷的平台，帮商家做选品、投放、供应链和客服自动化。
```

更好的收束方式：

```text
请用构建者模式帮我砍功能。先不要做平台，只判断第一版最小可展示的价值点是什么，哪些必须砍掉。
```

### 例子 3：从现象走到动作的短输出

```markdown
### 现象
- 想解决的是“跨境卖家不知道先看哪些选品信号”
- 第一版还没有明确展示对象，但用户希望一周内做出来

### 判断
- 当前最值得验证的不是整个平台，而是“卖家会不会因为一个更快的选品线索视图而愿意继续看第二次”
- 第一版必须极小，只保留一个可展示价值点

### 红旗
- 功能边界明显过大
- 展示对象还没锁定

### 动作
- 只做一个单页原型：聚合 3 类选品信号，给出一个简洁排序
- 先拿给 3 个卖家看，不做供应链和客服自动化

### 验证
- 如果卖家愿意继续追问“这个数据还能看什么”，说明价值点成立
- 如果他们觉得只是“看起来不错”，但不想继续用，说明第一版还没抓住核心价值
```

## 怎么用这些例子

- 如果你是用户，优先学“怎么把问题说得更可判断”
- 如果你是技能维护者，优先学“怎么把输出收束到动作和验证”
- 如果一个例子开始变成大段说明，优先删解释，不优先删判别锚点
