# 问题发现

## 这个框架什么时候用

当你不确定眼前的是不是真问题，而不是伪需求、弱需求或概念包装时，用问题发现。

它适合的场景：

- 用户自己也说不清这个问题到底痛不痛
- 你怀疑现在看到的是“想法”，不是“问题”
- 想在投入做方案前，先判断需求真实性
- 市场上有人在讨论，但你不确定用户会不会真的行动

它的目标不是“多找几个灵感来源”，而是判断：

- 这个问题是不是真的存在
- 对谁存在
- 出现得多频繁
- 有多痛
- 是否值得切换解决方案

## 什么时候别用

以下情况不要把问题发现当主框架：

- 用户和问题已经很清楚，当前卡点是怎么收敛第一版
- 你需要的是用户任务拆解，而不是问题真假判断
- 你已经确认问题存在，当前要排查的是产品链路或策略问题

这时应优先考虑：

- JTBD
- MVP / 减法思维
- 逆向思维

## 先问哪几个高判别问题

### 1. 这个问题发生在谁身上

- 是具体哪类人，不是泛人群
- 他们在什么场景下反复遇到这个问题

### 2. 用户现在怎么解决

- 有没有现成替代方案
- 是忍受、规避，还是已经在主动找别的方法

### 3. 这个问题有多痛

- 频率多高
- 成本多大
- 不解决会造成什么后果

### 4. 用户是否会行动

- 用户是否已经为此付出时间、金钱或注意力
- 用户是否愿意切换现有方案

### 5. 你的判断依据来自哪里

- 自己的经验
- 他人的抱怨
- 行为观察
- 搜索和社区
- 访谈或真实数据

## 哪些来源更值得信

问题发现可以从多个来源入手，但权重不同。

### 高价值来源

- 自己或身边人的真实重复烦恼
- 用户行为观察
- 目标用户的原话
- 用户已经存在的变通方案

### 中价值来源

- 社区讨论
- 搜索热度
- 竞品评论

### 低价值来源

- “最近这个词很火”
- AI 头脑风暴出来的点子
- 没有行为支撑的主观感觉

低价值来源可以启发，但不能直接当结论。

## 常见红旗

- 用户说“挺有意思”，但没有任何行动
- 问题只有抽象好处，没有具体场景
- 没人真的在为此付出时间或金钱
- 只有口头认可，没有行为证据
- 只有趋势词，没有目标用户
- 问题看起来很多人都有，但每个人都不够痛

## 什么时候该停

出现以下情况时，不要继续放大叙事，应停下来补验证：

- 说不清当前替代方案
- 说不清问题频率
- 说不清不解决的后果
- 目标用户不明确
- 所有依据都来自二手信息

停下后的动作：

- 找 3 到 5 个目标用户访谈
- 收集 5 条真实原话
- 观察 1 到 2 个用户如何完成当前任务
- 验证他们是否已经在找替代方案

## 最小验证动作

问题发现最重要的不是渠道，而是验证。

推荐最小验证：

1. 访谈 3 到 5 个目标用户，问他们最近一次遇到这个问题是什么时候
2. 让用户复述当前替代方案、成本和不满点
3. 记录他们是否已经在主动找别的做法

最小成功信号：

- 至少 2 到 3 个目标用户能描述高度相似的问题场景
- 用户已经在忍受明确成本，或已经尝试替代方案
- 用户对“解决这个问题”有行动意愿，而不只是口头兴趣

最小失败信号：

- 用户觉得“有更好也行，没有也无所谓”
- 问题只有新鲜感，没有迫切性
- 不同用户口中的问题完全不一致

## 输出时怎么写

推荐结构：

```markdown
## 问题发现判断

### 现象
- 当前问题描述：
- 当前目标用户：
- 当前依据来源：

### 判断
- 真实性：已知 / 推断 / 待验证
- 强度：高 / 中 / 低
- 频率：高 / 中 / 低
- 切换意愿：高 / 中 / 低

### 红旗
- [最关键的 1-3 个红旗]

### 动作
- [下一步最值得做的动作]

### 最小验证
- [访谈 / 观察 / 搜索 / 试探性验证]
- [成功 / 失败信号]
```

## 一个短例子

```markdown
### 现象
- 想解决的问题是“AI 时代每个人都需要第二大脑”
- 目标用户仍然非常宽泛
- 依据主要来自网上讨论热度

### 判断
- 真实性：待验证，不足以直接成立
- 强度：中低，像广义焦虑，不像单一高痛问题
- 频率：可能高，但具体触发场景不清
- 切换意愿：未知

### 红旗
- 目标用户过大
- 讨论热度被误当成需求强度
- 没说清当前替代方案

### 动作
- 收窄到一个具体人群，例如“知识型自由职业者”
- 访谈他们最近一次“信息过载”是如何发生的

### 最小验证
- 成功信号：至少 3 个用户明确表达高频信息过载，并已在尝试多种替代工具
- 失败信号：用户只是口头认同，但没有真实行为和切换意愿
```
