# 问题发现理论框架

## 核心概念

**确保问题是真实存在的，而不是伪需求。**

## 五个发现渠道

### 1. 自身痛点
从自己的烦恼中发现问题。

**优势**：
- 最真实，自己就是用户
- 可以快速验证解决方案

**方法**：
- 记录一周内的烦恼
- 留意自己抱怨的时刻
- 注意自己做的变通方案

**案例**：
- "我总是忘记带伞" → 天气提醒工具
- "我总是找不到上次看到的文章" → 稍后阅读工具

### 2. 他人口碑
从他人的抱怨和求助中发现问题。

**优势**：
- 问题真实存在
- 有现成的潜在用户

**方法**：
- 关注社交媒体上的抱怨
- 留意朋友/同事的求助
- 参与相关社区/论坛

**案例**：
- 朋友抱怨"记账太麻烦" → 简化记账工具
- 同事抱怨"总是错过会议" → 智能提醒工具

### 3. 观察记录
通过观察他人发现未被满足的需求。

**优势**：
- 发现用户自己都没意识到的需求
- 发现真实行为与口头表达的差异

**方法**：
- 跟踪目标用户一天
- 观察用户如何解决当前问题
- 记录用户的变通方案

**案例**：
- 观察到用户用纸笔记录 → 思考如何数字化
- 观察到用户的复杂工作流 → 思考如何简化

### 4. 网络搜索
从搜索趋势和热门内容中发现问题。

**优势**：
- 量化需求热度
- 发现广泛存在的痛点

**方法**：
- 搜索相关问题关键词
- 查看搜索建议/相关问题
- 分析问答平台的问题

**案例**：
- 搜索"如何XX"发现高频问题
- 分析知乎/Reddit相关问题的热度

### 5. AI 辅助
利用 AI 的分析和总结能力发现机会。

**优势**：
- 快速处理大量信息
- 发现人类难以察觉的模式

**方法**：
- 让 AI 总结用户评论的共性
- 让 AI 分析竞品的不足
- 让 AI 头脑风暴潜在问题

## 烦恼日记法

### 方法介绍

记录一周内的所有烦恼，筛选可解决的真实问题。

### 模板

```markdown
## 烦恼日记 - 第 [X] 天

### 烦恼 1
**时间**：[什么时候]
**场景**：[在什么情况下]
**烦恼**：[具体是什么]
**当前方案**：[现在怎么解决]
**痛点**：[当前方案哪里不爽]
**机会**：[是否有产品机会]

### 烦恼 2
...
```

### 一周后分析

1. **统计烦恼类型**：哪些类型重复出现？
2. **评估痛点强度**：哪些最让人难受？
3. **筛选机会**：哪些有产品化机会？
4. **验证需求**：找更多人确认是否有同样烦恼

## 问题验证五维度

发现潜在问题后，用五维度评估：

| 维度 | 问题 | 评估方法 |
|------|------|----------|
| **真实性** | 问题真的存在吗？ | 找5-10个目标用户确认 |
| **强度** | 痛点有多强烈？ | 用户愿意付出多少解决 |
| **频率** | 问题出现频率？ | 每天/每周/每月几次 |
| **迫切性** | 用户多迫切想解决？ | 用户是否主动寻找方案 |
| **可解决性** | 技术上可行吗？ | 评估技术实现难度 |

### 评估矩阵

```
│
│ 高迫切性
│  ┌─────────┬─────────┐
│  │ 优先    │  考虑    │
│  │ (高频+  │ (低频+  │
│  │ 高痛)   │ 高痛)   │
│  └─────────┴─────────┘
│  ┌─────────┬─────────┐
│  │  放弃   │  暂缓   │
│  │ (低频+  │ (高频+  │
│  │ 低痛)   │ 低痛)   │
│  └─────────┴─────────┘
│
└─────────────────────→ 痛点强度
   低                    高
```

## 常见陷阱

### 1. 伪需求
用户说想要，但实际不care。

**信号**：
- 用户说"挺好"但没有行动
- 用户承诺试用但没下文

**对策**：观察行为，不要只听用户说

### 2. 需求理解错误
理解的需求和真实需求不一致。

**信号**：
- 做出来的产品用户不用
- 用户使用方式和预期不同

**对策**：持续和用户沟通，快速迭代

### 3. 问题太小
问题存在，但市场太小。

**信号**：
- 只能找到很少的目标用户
- 用户不愿意付费

**对策**：评估市场规模，或将其作为更大产品的功能
