# JTBD

## 这个框架什么时候用

当你已经知道大致用户是谁、问题也大致存在，但还不清楚用户真正想完成什么任务时，用 JTBD。

它适合的场景：

- 用户嘴上说要某个功能，但你怀疑那不是底层任务
- 想把“功能需求”翻译成“用户想完成的事”
- 想理解当前替代方案为什么还能被忍受
- 想在做方案或定位前，先把用户任务拆清楚

它的目标不是产出一个概念定义，而是判断：

- 用户真正想完成的核心任务是什么
- 这个任务的功能层、情感层、社会层分别是什么
- 当前方案在哪一层最容易失手

## 什么时候别用

以下情况不要把 JTBD 当主框架：

- 用户是谁还说不清
- 问题是不是真的存在还没验证
- 当前更需要的是收敛第一版功能
- 当前主要任务是排查已有产品问题链路

这时应先做：

- 灵魂三问
- 问题发现
- MVP / 减法思维
- 必要时用诊断模式拆链路

## 先问哪几个高判别问题

### 1. 用户现在到底想完成什么事

- 不是想用什么功能
- 而是想在什么场景下达成什么结果

### 2. 用户现在怎么完成这件事

- 用什么工具
- 用什么流程
- 做了哪些变通

### 3. 当前方案最让用户不爽的点是什么

- 卡在哪一步
- 最容易放弃在哪一步
- 成本最高的是哪一步

### 4. 这件事除了功能目标，还有什么感受和身份目标

- 用户希望更安心、更高效，还是更显专业
- 这件事失败时，用户最在意失去什么

### 5. 如果不给这个产品，用户会怎么办

- 忍受
- 拼凑多个工具
- 找人帮忙
- 直接放弃

这个问题能逼出真正的替代方案和任务强度。

## 常见红旗

- 把“想要某功能”直接等同于用户任务
- 说得出功能列表，说不清任务闭环
- 只关注功能层，不看情感层和社会层
- 只听用户怎么说，不看用户怎么做
- 当前替代方案说不清
- 用户真正想避免的失败状态说不清

## 什么时候该停

出现以下情况时，不要继续往下画用户旅程，应先停下来补依据：

- 说不清当前替代方案
- 说不清用户在哪个场景下触发任务
- 说不清任务完成后的结果
- 只能说“用户想更方便”，说不出具体任务

停下后的动作：

- 找 3 个真实用户，问他们最近一次完成这件事是什么过程
- 记录他们现有流程中的卡点和变通方案
- 区分用户说想要的功能，和用户真正想完成的任务

## 最小验证动作

JTBD 的最小验证，不是写出三层需求，而是验证你对任务的理解是否贴近真实行为。

推荐最小验证：

1. 找 3 到 5 个目标用户，复盘他们最近一次完成该任务的全过程
2. 记录触发场景、当前工具、关键卡点和最终结果
3. 用一句话复述你理解的“核心任务”，看用户是否认同

最小成功信号：

- 多个用户能描述相似的触发场景和完成路径
- 你能清楚区分功能层、情感层、社会层
- 用户认可你复述的核心任务

最小失败信号：

- 每个用户说的都像不同问题
- 所谓任务其实只是功能偏好
- 用户不认同你提炼出的核心任务

## 输出时怎么写

推荐结构：

```markdown
## JTBD 判断

### 现象
- 当前目标用户：
- 当前任务场景：
- 当前替代方案：

### 判断
- 核心任务是什么：
- 功能层任务：
- 情感层任务：
- 社会层任务：

### 红旗
- [最关键的 1-3 个红旗]

### 动作
- [下一步最值得做的动作]

### 最小验证
- [找谁复盘]
- [验证什么]
- [成功 / 失败信号]
```

## 一个短例子

```markdown
### 现象
- 用户是刚开始做付费 SaaS 的独立开发者
- 现在用飞书文档、邮件和表格拼凑管理反馈

### 判断
- 核心任务不是“拥有一个反馈工具”
- 而是“在分散反馈里快速判断什么值得做，并在需要时能回看依据”
- 功能层任务：收集、归类、回看反馈
- 情感层任务：减少混乱和漏掉重要反馈的焦虑
- 社会层任务：显得自己在做有依据的产品决策

### 红旗
- 现在还没说清用户最痛的是收集、整理还是回看
- 如果不区分这三步，产品很容易做成大杂烩

### 动作
- 访谈 3 个独立开发者，复盘他们最近一次整理反馈的完整过程

### 最小验证
- 成功信号：多个用户都表达“回看和决策依据”是核心痛点
- 失败信号：每个人最痛的步骤完全不同，说明任务仍未收敛
```
