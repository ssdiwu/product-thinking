# multi-agent 模式详细指南

> 基于 ai-team 的 multi-agent 模式改造

## 核心理念

**专家协作**: product-thinking 作为顾问,派发专家 Agent 并行执行分析任务,汇总结果并提供策略建议。

```
product-thinking (顾问)
    ↓ 派发任务
专家 Agent 1 (JTBD)  ←→  专家 Agent 2 (MVP)  ←→  专家 Agent 3 (场景)
    ↓ 汇总结果
product-thinking (综合分析)
```

## 专家 Agent 体系

### 1. 产品战略专家 (Product Strategy Expert)

**职责**: JTBD 分析和用户画像

**输入**:
- 灵魂三问的回答
- 用户画像(如果是 B2B)

**输出**:
- 文档: `phases/02-jtbd.md`
- 三层需求分析(功能层/情感层/社会层)
- 用户旅程图(Mermaid)
- 核心发现总结

**分析框架**:
1. 用户"雇佣"产品来完成什么工作?
2. 功能层需求: 用户想完成什么具体任务?
3. 情感层需求: 用户希望获得什么感受?
4. 社会层需求: 用户希望在他人眼中是什么形象?
5. 绘制用户旅程图: 从触发 → 行动 → 结果

**验收标准**:
- [ ] 三层需求都已分析
- [ ] 用户旅程图已生成
- [ ] 核心发现已总结(3-5 个)
- [ ] 文档格式符合模板

**参考文档**: `references/framework/jtbd.md`

---

### 2. MVP 设计专家 (MVP Design Expert)

**职责**: 功能分层和优先级排序

**输入**:
- JTBD 分析结果
- 用户的核心任务

**输出**:
- 文档: `phases/03-mvp.md`
- P0/P1/P2 功能分层
- MVP 验证指标
- 功能分层图(Mermaid)

**分析框架**:
1. 核心假设是什么?(需要验证的最关键假设)
2. P0 功能: 验证核心假设的最小功能集
3. P1 功能: 增强体验但非必需
4. P2 功能: 锦上添花
5. 验证周期: 多久可以验证核心假设?

**验收标准**:
- [ ] 核心假设已明确
- [ ] P0/P1/P2 功能已分层
- [ ] MVP 验证指标已定义
- [ ] 功能分层图已生成
- [ ] 文档格式符合模板

**参考文档**: `references/framework/mvp.md`

### 3. 场景分析专家 (Scenario Analysis Expert)

**职责**: 场景匹配和切入策略

**输入**:
- JTBD 分析结果
- MVP 功能列表

**输出**:
- 文档: `phases/04-scenarios.md`
- 场景匹配矩阵
- 场景优先级排序
- 场景分析图(Mermaid)

**分析框架**:
1. 列举所有可能的使用场景
2. 评估每个场景的匹配度(用户需求强度 × 产品能力)
3. 评估每个场景的竞争强度
4. 选择最佳切入场景(高匹配度 + 低竞争强度)
5. 制定场景切入策略

**验收标准**:
- [ ] 场景匹配矩阵已生成
- [ ] 场景优先级已排序
- [ ] 最佳切入场景已选定
- [ ] 场景分析图已生成
- [ ] 文档格式符合模板

**参考文档**: `references/framework/scenarios.md`

   - 为什么会有这个问题?
   - 为什么会这样?
   - 为什么会这样?
   - 为什么会这样?
   - 为什么会这样?
3. 绘制问题分析树

**验收标准**:
- [ ] 问题真实性已验证

- [ ] 问题分析树已生成
- [ ] 文档格式符合模板

### 5. 综合分析专家 (Synthesis Expert)

**职责**: 汇总所有分析结果,制定产品策略

**输入**:
- 所有分析工具的文档

**输出**:
- 文档: `analysis/综合分析.md`
- 核心发现汇总
- 关键洞察(3-5 个)
- 产品策略建议
- 风险评估
- 资源需求
- 行动计划

**分析框架**:
1. 读取所有分析文档
2. 提取核心发现
3. 汇总关键洞察
4. 制定产品策略:
   - 目标用户画像
   - 产品定位(一句话定位)
   - 功能规划(MVP → 功能扩展 → 生态完善)
   - 场景切入策略
5. 评估风险(高/中/低风险项)
6. 规划资源(人力/技术/时间)
7. 制定行动计划(立即/短期/中期)

**验收标准**:
- [ ] 所有分析工具的核心发现都已汇总
- [ ] 关键洞察已提炼(3-5 个)
- [ ] 产品策略已制定
- [ ] 风险评估已完成
- [ ] 资源需求已明确
- [ ] 行动计划已列出
- [ ] 文档格式符合模板

**参考文档**: `references/03-execution.md`

---

### 6. 质量审核专家 (Quality Review Expert)

**职责**: 文档完整性和质量检查

**输入**:
- 所有生成的文档

**输出**:
- 文档: `reviews/completeness-check.md`
- 综合评分
- 改进建议

**审核维度**:
1. 文档完整性(30%): 所有必需的文档都已生成
2. 内容质量(30%): 分析深度、逻辑一致性、数据支撑
3. 可视化图表(20%): 图表完整性、语法正确性
4. 验收标准(20%): 所有验收标准都已满足

**验收标准**:
- [ ] 综合评分 >= 80 分
- [ ] 所有高优先级改进项已列出
- [ ] 改进建议具体可执行
- [ ] 文档格式符合模板

**参考文档**: `references/04-review.md`

## 并发控制

### 可配置并发数

**配置方式**（按优先级从高到低）：

1. **启动参数**（最高优先级）：
   ```bash
   /product-design --concurrency 5
   ```

2. **project.yaml 配置**：
   ```yaml
   concurrency: 3
   ```

3. **默认值**（向后兼容）：
   ```
   默认值: 3
   ```

**配置验证规则**：
- 最小值：2（主 Agent + 1 个专家 Agent）
- 最大值：10（避免过度并发导致 429）
- 无效值：使用默认值 3 并警告用户

**示例**：

```yaml
# project.yaml
project_name: "智能客服系统产品设计"
concurrency: 5  # 配置并发数为 5
analysis_tools:
  - jtbd
  - mvp
  - scenarios
```

### 核心约束

```
并发不变量(必须永远为真):
- 同一时刻活跃的 Agent 总数 <= N (N 为配置的并发数,默认 3)
- 主 Agent (product-thinking) 始终占用 1 个并发槽位
- 专家 Agent 最多使用 (N-1) 个槽位
- 收到 429 错误后,立即降级为串行模式
```

### 批次执行引擎

当分析任务数量超过可用槽位时,自动进入批次模式:

```
输入: tasks = [JTBD, MVP, 场景, 诊断]
配置: concurrency = 5 (可用专家 Agent 槽位 = 4)

计算:
  可用专家 Agent 槽位 = concurrency - 1 = 4
  批次数 = ceil(4 / 4) = 1

执行:
  批次 1: [JTBD, MVP, 场景, 诊断] → 并行执行 → 等待全部完成

  product-thinking 在批次完成后做结果汇总和质量检查
```

**示例 2**（默认并发数 3）：

```
输入: tasks = [JTBD, MVP, 场景, 诊断]
配置: concurrency = 3 (默认值)

计算:
  可用专家 Agent 槽位 = 3 - 1 = 2
  批次数 = ceil(4 / 2) = 2

执行:
  批次 1: [JTBD, MVP] → 并行执行 → 等待全部完成
  批次 2: [场景, 诊断] → 并行执行 → 等待全部完成

  product-thinking 在每个批次之间做结果汇总和质量检查
```

### 429 错误处理

```
429 发生
  │
  ▼
立即暂停所有待发专家 Agent(不中断已运行的)
  │
  ▼
等待已运行的专家 Agent 自然完成
  │
  ▼
进入冷却期(默认 60 秒)
  │
  ▼
发送 1 个探测 Agent(最轻量任务)
  │
  ├── 成功 → 恢复到降级模式(并发数 = 1),逐步增加
  │           连续 3 次成功 → 恢复正常模式
  │
  └── 失败 → 延长冷却期(翻倍,最大 5 分钟)
              进入串行模式
```

## 工作流详解

### 阶段 1: Kickoff

**执行步骤**:
1. 完成灵魂三问
2. 判断 B2C/B2B
3. 选择分析工具
4. 生成分析框架图
5. 初始化 `.product-thinking/` 目录

**产出**:
- `phases/01-kickoff.md`
- `project.yaml`

---

### 阶段 2: Analysis

**执行步骤**:
1. 按依赖关系排序分析任务
2. 划分批次(每批次最多 2 个任务)
3. 为每个任务生成任务包
4. 通过 Task 工具派发专家 Agent
5. 收集产出,更新项目状态
6. 遇到依赖阻塞时: 优先重排序,无法重排则等待

**任务包结构**:

```markdown
# 任务包: JTBD 分析

## 角色
你是产品战略专家,专注于 JTBD 分析和用户画像。

## 任务
分析用户真正想完成的任务,提取三层需求。

## 输入
- 灵魂三问的回答
- 用户画像

## 输出
- 文档: `phases/02-jtbd.md`
- 三层需求分析(功能层/情感层/社会层)
- 用户旅程图(Mermaid)
- 核心发现总结

## 分析框架
1. 用户"雇佣"产品来完成什么工作?
2. 功能层需求: 用户想完成什么具体任务?
3. 情感层需求: 用户希望获得什么感受?
4. 社会层需求: 用户希望在他人眼中是什么形象?
5. 绘制用户旅程图: 从触发 → 行动 → 结果

## 验收标准
- [ ] 三层需求都已分析
- [ ] 用户旅程图已生成
- [ ] 核心发现已总结(3-5 个)
- [ ] 文档格式符合模板

## 参考文档
- `references/framework/jtbd.md`
- `references/templates/jtbd-template.md`

## 约束
- 必须使用 Mermaid 生成用户旅程图
- 必须遵循文档模板格式
- 必须满足所有验收标准
```

**强制检查点**:
- **任务完成时** → 立即更新 `project.yaml` 状态文件
- **批次完成后** → 记录完成度扫描
- **全部完成时** → 自动触发 Execution 阶段

---

### 阶段 3: Execution

**执行步骤**:
1. 派发综合分析专家 Agent
2. 汇总所有分析结果
3. 制定产品策略
4. 生成综合分析报告

**产出**:
- `analysis/综合分析.md`

---

### 阶段 4: Review

**执行步骤**:
1. 派发质量审核专家 Agent
2. 多维度评分
3. 生成改进建议
4. 生成审核报告

**产出**:
- `reviews/completeness-check.md`

---

### 阶段 5: Closing

**执行步骤**:
1. 汇总所有交付物
2. 生成交付状态矩阵
3. 输出分析流程可视化
4. 记录已确认决策和待跟进项
5. 生成最终报告

**产出**:
- `reports/final-report.md`

## 关键优势

### 1. 并行执行,速度快

多个专家 Agent 并行执行分析任务,大幅缩短分析时间。

### 2. 专家视角,质量高

每个专家 Agent 专注于一个分析维度,分析深度更高。

### 3. 自动协调,无需人工干预

product-thinking 自动协调专家 Agent 的依赖关系和执行顺序。

### 4. 降级保护,稳定性强

遇到 429 错误时自动降级,确保任务一定能完成。

## 最佳实践

### 1. 任务包要完整

每个任务包必须包含:
- 角色定义(专家的身份)
- 任务描述(要做什么)
- 输入输出(需要什么,产出什么)
- 分析框架(如何分析)
- 验收标准(如何验证)
- 参考文档(在哪里找资料)
- 约束条件(必须遵守的规则)

### 2. 依赖关系要明确

在派发任务前,明确依赖关系:
- JTBD 分析 → 无依赖
- MVP 功能审视 → 依赖 JTBD 分析
- 场景应用分析 → 依赖 JTBD 分析 + MVP 功能审视
- 问题发现诊断 → 依赖 JTBD 分析

### 3. 批次划分要合理

按依赖关系划分批次:
- 批次 1: [JTBD 分析] (无依赖,优先执行)
- 批次 2: [MVP 功能审视, 问题发现诊断] (依赖批次 1)
- 批次 3: [场景应用分析] (依赖批次 1 + 批次 2)

### 4. 监控 429 错误

在派发任务时,监控 429 错误:
- 如果收到 429 → 立即暂停待发任务
- 等待已运行任务完成
- 进入冷却期
- 发送探测 Agent
- 根据探测结果决定恢复或降级

## 常见问题

### Q: multi-agent 模式和 codex-assisted 模式有什么区别?

**A**:
- **multi-agent**: AI 自动派发专家 Agent 并行执行,适合快速迭代
- **codex-assisted**: AI 生成任务清单,用户用 Codex 执行,适合精确控制

**选择建议**:
- 需要快速迭代 → multi-agent
- 需要精确控制 → codex-assisted

### Q: 如果专家 Agent 执行失败怎么办?

**A**: product-thinking 会:
1. 检查失败原因
2. 如果是 429 错误 → 进入降级模式
3. 如果是其他错误 → 重试(最多 2 次)
4. 如果重试失败 → 降级为 sequential 模式,自己执行

### Q: 如何调整并发数?

**A**: 并发数支持三种配置方式（按优先级从高到低）：

1. **启动参数**（推荐）：
   ```bash
   /product-design --concurrency 5
   ```

2. **project.yaml 配置**：
   ```yaml
   concurrency: 5
   ```

3. **默认值**：3（主 Agent + 2 个专家 Agent）

**配置建议**：
- Claude Pro 用户：建议 3-5
- 企业用户（更高速率限制）：可以设置 5-10
- 测试环境：建议 2（最小值）

**验证规则**：
- 最小值：2（主 Agent + 1 个专家 Agent）
- 最大值：10（避免过度并发）
- 无效值：自动使用默认值 3 并警告

### Q: 如果某个专家 Agent 的产出质量不高怎么办?

**A**: Review 阶段会检查质量,如果不合格:
1. 生成改进建议
2. 重新派发该专家 Agent
3. 传递改进建议作为额外约束
